<objective>
Close the two blocking/high-priority gaps from 05-VERIFICATION.md (CR-01) and 05-REVIEW.md (WR-02) found in Phase 5 (PDF Export):

1. **CR-01 (blocker):** `pdfExport.ts` hardcodes `doc.addImage(..., 'PNG', ...)` regardless of the actual logo encoding. When a trainer uploads a JPEG logo, `downscaleImageBlob()` currently preserves JPEG encoding, producing a `data:image/jpeg` URI that jsPDF's `'PNG'`-forced decoder cannot parse — PDF export silently fails for any club with a JPEG logo.
2. **WR-02 (bundled per user decision):** Logos are drawn into a fixed 25mm x 20mm box regardless of their actual aspect ratio, stretching/squashing any non-1.25:1 logo even though `downscaleImageBlob()` deliberately preserves aspect ratio at upload time.

**Locked fix approach (per user decision, do not re-litigate):**
- Fix CR-01 by normalizing all uploaded images to PNG **at upload time** in `downscaleImageBlob()` (`imageDownscale.ts`) — not by format-sniffing in `pdfExport.ts`, not by restricting uploads to PNG-only. This makes the existing hard-coded `doc.addImage(logoData, 'PNG', ...)` calls in `pdfExport.ts` correct by construction; `pdfExport.ts` needs NO format-detection code.
- Fix WR-02 by computing a "contain" (fit-within-box, preserve-aspect-ratio) placement in `pdfExport.ts` using the image's actual pixel dimensions, instead of assuming the image fills the full 25x20mm box.

Purpose: Unblock 05-VERIFICATION.md's `gaps_found` verdict (4/5 must-haves -> 5/5) so Phase 5 can be re-verified as fully achieved.
Output: JPEG-uploaded logos export correctly as valid, undistorted PDFs; unit tests cover both the format-normalization and aspect-ratio-preservation behaviors.
</objective>
