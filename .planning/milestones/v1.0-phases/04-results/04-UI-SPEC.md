## Spacing Scale

**Inherited from Phase 1–3 — no changes:**

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps, inline badge padding |
| sm | 8px | Compact element spacing |

| lg | 24px | Section padding, grid gap (desktop) |
| xl | 32px | Layout gaps between shell regions |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level spacing |

**Phase 4 exceptions:**
- Podium rank badge: **24×24px** minimum (small, purely visual numeral badge, not a touch target — exempt from the 44px touch-target floor since it is not independently interactive).
- Results table cells: visually consistent, spacing-grid-compliant with Phase 3's `ScoreTable.svelte` — data cells `px-2 py-2 md:px-3 md:py-2` (8px/8px mobile, 12px/8px desktop), header cells `p-2 md:p-4` — for visual consistency between the two opaque, high-contrast data tables in the app, without propagating `ScoreTable.svelte`'s pre-existing non-grid-compliant mobile padding.
- Multi-class grid gap: **md (16px)** on tablet (`md`–`lg`), **lg (24px)** on desktop (`xl`+).
- Reset action: **44×44px minimum** touch target (same floor as all primary actions).
- Class card (per-class results container): **md (16px)** internal padding on phone/tablet, **lg (24px)** on desktop — matches Phase 2's card-padding convention.

### In-Progress Marker (D-03 — Claude's Discretion resolved)

A **muted superscript asterisk** immediately after the Gesamt (total) value for any shooter whose sum is still growing (not every arrow of every configured round/passe entered yet) — visually distinct from finished rows without needing a new color or new type size:

- Character: `*` (not an icon — keeps the table's dense, high-contrast character rendering fast and simple).
- Color: muted neutral — Light: `#94A3B8` (slate-400) · Dark: `#64748B` (slate-500) — deliberately NOT the destructive or accent color, since an in-progress shooter is not an error state.
- Placement: directly after the Gesamt numeral, e.g. `187*`, no space.
- Screen-reader text: `aria-label` on the cell (or a visually-hidden `<span>`) reading "Ergebnis noch nicht vollständig" (result not yet complete), so the marker isn't silent for assistive tech.
- **Legend caption:** a single Label-role (14px/400) line below each class's table (or once, below the last table on the page): `* Ergebnis noch nicht vollständig — Erfassung läuft.` (Result not yet complete — scoring in progress.) — rendered once per table for phone/dropdown mode, or once per visible class card in grid mode.

## Results Table Structure (RES-01, RES-02, D-01–D-07)

### Columns (fixed order, NOT user-sortable)

Unlike Phase 3's `ScoreTable` (which is interactively sortable per SCORE-04), the Results table's sort order is **fixed**: rank ascending (i.e., highest total first) — this is the entire point of the view, so no click-to-sort headers are built here. Columns:

1. **Rang** (Rank) — 1-2-2-4 shared-rank/skip-next numeral, with podium badge styling for ranks 1–3 (see above).
2. **Name** — shooter's name.
3. **Schießplatz** (Line) — shooter's assigned line number, shown as plain text (`row.line ?? '—'`), same fallback convention as `ScoreTable.svelte`. Hidden below `md` (768px) per the same `hidden md:table-cell` compaction pattern used for Score Entry's Klasse column (quick task 260705-p25) — line assignment is secondary information on phone.
4. **Gesamt** (Total) — tournament-wide sum across ALL configured rounds/passes (not per-passe, unlike Score Entry's "Summe") — right-aligned, bold (600 semibold), with the in-progress marker appended where applicable.

**No "Klasse" column** inside a per-class table — the class name is the section/card heading above the table (grouping already conveys it; repeating it in a column would be redundant, unlike Score Entry's cross-class table).

### Ranking Computation (RES-02, D-02)

- Every registered shooter appears, regardless of completion status (D-02) — one sorted list per class, no separate "in progress" bucket.
- Tournament-wide sum = sum of every recorded arrow across every round × passe × arrow index for that shooter (X = 10, M = 0, per the existing `arrowScoreValue` convention in `scoreCompletion.ts` — the new ranking function should reuse `arrowScoreValue`, not reimplement value mapping).
- Rank assignment: standard competition ranking — sort descending by sum; shooters with equal sums share the same rank number; the next distinct sum's rank is the shooter's 1-based position in the sorted list (i.e., "1-2-2-4" skip-next), not "1-2-2-3".
- Sort tiebreak among equal-sum shooters for **row order only** (rank number is identical either way): alphabetical by name, for deterministic, non-flickering rendering as `liveQuery` re-fires.

### Table Rendering

- **Background:** fully opaque — Light `#FFFFFF`, Dark `#1E293B` (per D-06, Phase 1 D-11 — no glass treatment on the table itself, only the optional card frame around it).
- **Borders:** Light `1px solid #E2E8F0` (slate-200), Dark `1px solid #475569` (slate-600) — identical to `ScoreTable.svelte`.
- **Row striping:** optional, same convention as `ScoreTable.svelte` if adopted for readability.
- Table is wrapped in `overflow-x-auto` as a safety net (same as `ScoreTable.svelte`), though the 4-column Results table is narrower than Score Entry's and should rarely need to scroll.

## Multi-Class Responsive Layout (RES-03, RES-04, D-04, D-05)

### Class Ordering (D-04)

Classes are ordered **alphabetically by class name** wherever multiple appear — both in the tablet/desktop grid and in the phone dropdown's option list.

### Phone (<768px / below `md`) — RES-03

- **Single class visible at a time**, selected via a **native `<select>` dropdown** (same convention as Score Entry's Runde/Passe selectors and Setup's dropdowns — no third-party dropdown library).
- Dropdown label: "Klasse" (Label role, 14px), positioned above the select, same layout pattern as `RoundPasseSelector.svelte`.
- Dropdown options: alphabetical class-name list (D-04); default selection: first class alphabetically.
- Below the dropdown: the single selected class's results table + in-progress legend (if applicable).
- Dropdown container may be glass-treated (per Phase 1 D-11 scope — dropdowns/controls are UI chrome, not the data table itself), matching Setup/Score-Entry control styling.

### Tablet/Desktop (≥768px / `md` and above) — RES-04, D-05

**Responsive grid** of per-class cards (Claude's discretion on exact breakpoints, resolved below — consistent with the app's existing `md` = 768px nav-switch breakpoint and `xl` = 1280px sidebar-rail breakpoint):

| Breakpoint | Columns |
|------------|---------|
| `md` to `lg` (768px–1023px, narrow tablet) | 1 column (`grid-cols-1`) |
| `lg` to `xl` (1024px–1279px) | 2 columns (`lg:grid-cols-2`) |
| `xl`+ (≥1280px, wide desktop) | 3 columns (`xl:grid-cols-3`) |

- Each grid cell is a card (`GlassCard.svelte` framing, per-class Heading-role title + the opaque results table inside, per D-06).
- Grid gap: **md (16px)** at `md`/`lg`, **lg (24px)** at `xl`+ (see Spacing Scale).
- If only 1 class exists, the grid naturally renders as a single card regardless of breakpoint — no special-casing needed.
- Cards render in the alphabetical class order (D-04), filling the grid left-to-right, top-to-bottom (standard CSS grid auto-flow).

## Reset Flow (RES-05, D-08, D-09, D-10)

### Placement (D-08)

The "Neues Turnier starten" action lives **on the Results view itself**, below the per-class results grid/dropdown — no new nav section or settings area.

### Button

- Label: **"Neues Turnier starten"** (verb + noun, matches the copywriting contract's CTA pattern).
- Style: **destructive** treatment (same red token as `ConfirmDialog`'s `destructive` prop, per the Color contract above) — NOT accent-colored, since this is explicitly a data-clearing action, not a primary/positive action.
- Icon: `RotateCcw` (20px, `@lucide/svelte`), leading the label text.
- Size: 44×44px minimum touch target; full-width on phone, natural width on tablet/desktop (matches Phase 2/3 button-width conventions).

### Confirmation (D-09)

Reuses `ConfirmDialog.svelte` verbatim — same props/pattern already used for "Abschließen" (Phase 3) and preset delete/overwrite (Phase 2):
- `title`: **"Neues Turnier starten?"**
- `body`: **"Alle Schützen und Punkte werden gelöscht. Klassen, Schießplätze und Runden/Passen bleiben erhalten. Diese Aktion kann nicht rückgängig gemacht werden. Fortfahren?"** (All shooters and scores will be deleted. Classes, shooting lines, and rounds/passes are retained. This action cannot be undone. Continue?)
- `confirmLabel`: **"Ja, zurücksetzen"**
- `cancelLabel`: **"Abbrechen"**
- `destructive`: `true`

### On Confirm (D-10)

1. `db.shooters` and `db.scores` tables are cleared (`.clear()`); `db.classes`, `db.shootingLines`, `db.rounds`, `db.presets` are left untouched.
2. Modal closes.
3. Results view now shows the empty state (see Copywriting Contract) since no shooters remain.
4. A brief confirmation message displays: **"Turnier zurückgesetzt. Klassen, Schießplätze und Runden/Passen sind erhalten."** (toast or inline message, non-blocking, auto-dismiss or manually dismissible — same convention as Phase 2/3 success toasts).

## Destructive-Edit Guard (RES-06, D-11, D-12)

### Trigger Condition (D-12)

The guard activates **only when `isFinalized === true`** (the same boolean Phase 3's `ScoreEntry.svelte` computes, per 03-CONTEXT.md D-09/D-10's permanent-lock semantics) — NOT merely "any score exists." While a tournament is live/in-progress, Setup and Registration remain fully editable, consistent with D-01's "results are viewable anytime, mid-tournament" decision.

### Guarded Controls (D-11 + Claude's Discretion on class deletion)

Implemented as **disabled controls with an inline message** — never an "intercepted by dialog at click time" pattern:

| View | Guarded control(s) | Guard treatment |
|------|--------------------|------------------|
| Registration (`Registration.svelte`) | Delete-shooter button (per shooter row) | `disabled`, reduced opacity (50%), `cursor: not-allowed`, `aria-disabled="true"` |
| Setup (`Setup.svelte` / `SetupRounds.svelte`) | Rounds/passes config form (all inputs), shooting-line count input | Same disabled treatment as above, applied to the whole form's inputs |
| Setup (`ClassForm.svelte` / class list) | Delete-class button (per class row) | Same disabled treatment — **guarded conservatively per Claude's discretion**, since deleting a class after finalization would orphan finalized shooter records exactly as a shooter deletion would |

### Inline Message

Exact copy (D-11's example, locked as the final wording): **"Turnier abgeschlossen — Zurücksetzen, um zu ändern."** (Tournament finalized — reset to make changes.)

- Positioned directly adjacent to (below or beside) each disabled control/form, Body role (16px/400), muted neutral color (slate-500/slate-400) — not destructive-red, since this is informational, not an error.
- Rendered via `role="status"` or plain text (no need for `aria-live` — the message is present from initial render whenever the guard is active, not injected reactively mid-interaction).
- Does NOT include a direct link/button to the Results view's reset action from this message in v1 (keeps scope minimal — the trainer navigates to Ergebnisse manually); Claude may add a lightweight text-only pointer if trivial, but a full inline CTA button is out of scope.

### Un-guarded (before finalization)

No visual change at all to Setup/Registration controls while `isFinalized === false` — this guard is new UI surface introduced entirely by Phase 4 and must not alter Phase 2/3's existing appearance for the live-tournament state.

## Layout & Interaction

### Results View Structure

**Top section:** Page/section heading "Ergebnisse" (Heading role — or omitted if the nav bar already labels the section, consistent with how Phase 2/3 views handle their own heading vs. relying on nav).

**Class selector (conditional by breakpoint):**
- Phone (<768px): single "Klasse" dropdown (RES-03).
- Tablet/desktop (≥768px): no selector — all classes render simultaneously in the responsive grid (RES-04).

**Main content:**
- Phone: one class's results table + in-progress legend (if any incomplete shooters).
- Tablet/desktop: grid of per-class cards, each with class-name heading + results table + legend.
- Empty state (no shooters registered at all): centered Body-role message, no table rendered (see Copywriting Contract).

**Bottom section:**
- "Neues Turnier starten" (reset) button, destructive-styled, full-width on phone.

### Live-Updating (D-01)

The whole view uses `liveQuery()` + `$derived` (same established pattern from Phases 2–3) over `db.shooters`, `db.classes`, and `db.scores` — no manual refresh action. Results are visible and update in real time whether the tournament is mid-entry or finalized; there is no gated/locked/placeholder state for this view based on finalization status (only the guard in Setup/Registration is finalization-gated, not this view itself).

### Interaction Flow

1. **On view entry:** Compute per-class ranked lists from current `db.shooters`/`db.scores`/`db.classes` state. If no shooters exist, show empty state. Otherwise render the phone dropdown or tablet/desktop grid depending on viewport.
2. **Mid-tournament (not finalized):** Rows update live as scores are entered elsewhere (Score Entry view); in-progress shooters show the asterisk marker; ranks recompute reactively.
3. **Post-finalization:** Same live view continues to render (no locked/read-only visual change to the Results view itself — RES-06's guard applies to Setup/Registration, not here). Reset button remains available regardless of finalization state (a trainer may reset a tournament before or after finalizing, per RES-05's plain wording).
4. **Reset:** Trainer clicks "Neues Turnier starten" → `ConfirmDialog` appears → on confirm, `db.shooters.clear()` and `db.scores.clear()` run → view re-renders to empty state → success message shows.
5. **Guard (elsewhere):** If the trainer navigates to Setup or Registration while `isFinalized === true`, delete-shooter/delete-class buttons and the rounds/passes form are disabled with the inline guard message; navigating there while not yet finalized shows the normal, fully-editable Phase 2/3 UI.

## Component Architecture (Svelte 5)

### New Components to Build

1. **Results.svelte** — Main view/page component. Orchestrates class grouping, breakpoint-based selector-vs-grid rendering, reset flow. Reuses `ConfirmDialog`.
2. **ResultsTable.svelte** — Per-class ranked table (Rang, Name, Schießplatz, Gesamt). Props: `rows` (ranked, with `rank`, `isComplete` per row), `showLineColumn` (responsive). Purely presentational — no sort-click handlers (fixed order).
3. **ClassSelector.svelte** — Phone-only native `<select>` for class choice. Props: `classes` (alphabetical), `selectedClassId`. Emits `classChanged(classId)`.

### New Pure Utility

- A tournament-wide ranking function (e.g. `computeClassRankings` in a new or extended `src/lib/utils/` module) following `scoreCompletion.ts`'s established style (plain functions, no framework imports, comment banner referencing RES-01/RES-02/D-02): sums every round × passe × arrow per shooter (reusing `arrowScoreValue`), groups by class, sorts descending, assigns shared/skip-next ranks, and flags `isComplete` per shooter using the same cell-completeness logic pattern as `areAllScoresEntered` (scoped to one shooter's full score set rather than the whole tournament).

### State Management

- **Class grouping / ranking:** `const rankings = $derived(computeClassRankings(shooters, classes, scores));` — reactive over `liveQuery` results, same idiom as Phases 2–3.
- **Selected class (phone only):** `let selectedClassId = $state<number | null>(null);` — defaults to the first alphabetical class once data loads.
- **isFinalized (cross-view):** Read the same way `ScoreEntry.svelte` already computes it in Phase 3; expose it (e.g. via a small shared utility or a `liveQuery` on `db.scores`) so `Setup.svelte`/`Registration.svelte` can also read it for the RES-06 guard — this is a new cross-view integration point Phase 4 introduces, not a new UI pattern.

## Accessibility Requirements

- Podium badges: `aria-label` on the Rang cell stating the plain rank (e.g. `aria-label="Rang 1"`) in addition to the visual badge — badge color alone must not be the only signal (color-blind safe: numeral is always visible as text inside the badge).
- In-progress marker: `aria-label`/visually-hidden text per row, per the In-Progress Marker section above — never color/asterisk alone.
- Phone class dropdown: standard `<label for>` association, same as other native selects in the app.
- Reset button: no ambiguity needed (full text label, not icon-only), but still carries `aria-label` matching its visible text for consistency with the rest of the app's button conventions.
- Guard inline messages: plain text with sufficient contrast (slate-500/400 on the app's opaque/glass surfaces meets WCAG AA at 16px), not solely conveyed via disabled-control styling.

## Error Handling & Edge Cases

- **Reset failure:** If `db.shooters.clear()`/`db.scores.clear()` throws (e.g., storage error), show `results.resetError` in a non-blocking message div (destructive red), same convention as Phase 3's autosave-failure handling — the confirmation modal has already closed by this point (optimistic close), so this is a post-hoc error surface, not a blocking retry loop.
- **No shooters in a class:** A class with zero registered shooters does not render an empty table — it is simply absent from the grid/dropdown until a shooter is registered to it (consistent with classes being configured independently of registration in Phase 2).
- **All shooters incomplete:** Every row shows the in-progress marker; ranks still compute normally over partial sums (D-02) — this is expected, not an error state.
- **Tied sums across the whole class (e.g., all shooters at 0):** All shooters share rank 1; no podium accent conflict since D-07 already establishes "every row sharing a rank gets that rank's accent."
