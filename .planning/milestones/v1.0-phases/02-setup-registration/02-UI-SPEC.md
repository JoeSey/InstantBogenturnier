# Phase 2 — UI Design Contract

> Visual and interaction contract for Setup & Registration. Extends Phase 1's glassmorphism system. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none — same as Phase 1: no shadcn/ui (React-only); custom Svelte 5 components with Tailwind CSS 4 |
| Preset | not applicable |
| Component library | none (custom Svelte 5 components) — extends `GlassCard.svelte` from Phase 1 for Setup/Registration forms and preset list items |

## Spacing Scale

**Inherited from Phase 1 — no changes:**

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps, inline badge padding |
| sm | 8px | Compact element spacing (nav icon/label gap, form input gap) |
| md | 16px | Default element spacing, form field spacing, card/panel internal padding |
| lg | 24px | Section padding, form section breaks |
| xl | 32px | Layout gaps between major shell regions |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level spacing |

Exceptions (Phase 2 specific):
- Form buttons: **44×44px minimum** touch target for phone use (same as nav from Phase 1).
- Glass-card form containers: md (16px) internal padding on phone, lg (24px) on tablet/desktop (responsive via Tailwind breakpoints).
- Preset list items (glass cards): md padding, md gap between items.
- Auto-assignment preview modal: lg padding internally, centered content max-width ~520px.

**Inherited from Phase 1 — no changes:**

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Label | 14px | 400 (regular) | 1.4 |
| Body | 16px | 400 (regular) | 1.5 |
| Heading | 20px | 600 (semibold) | 1.2 |
| Display | 28px | 600 (semibold) | 1.2 |

Exactly 4 sizes, 2 weights (400 / 600) — no other combinations permitted.

**Phase 2 usage mapping:**
- **Label** — form field labels (Alter, Bogentyp, Entfernung, Name des Schützen, etc.), preset list item meta-text (created date, shooter count), modal labels.
- **Body** — form helper text, empty state copy, confirmation prompt text, preset name display.
- **Heading** — section headers (Klassen, Schützen, Presets), modal titles, preview headers.
- **Display** — page-level screens (e.g., "Einrichtung" when first entering Setup view).

**Accent reserved for (Phase 2 expansions):**
1. Active nav item (inherited from Phase 1).

3. Focus rings on form inputs (`:focus-visible` outline).
4. Confirmation action buttons in modals (e.g., "Ja, Presets ersetzen" in the import confirmation modal).

Destructive actions (Phase 2 introductions):
- Delete preset buttons → destructive red.

## Layout & Interaction

### Setup View (Section: Einrichtung)

**Structure:** Sequential workflow form, top-to-bottom, with four collapsible/expandable card sections:

1. **Classes Card** — Form to add a new class (age-group dropdown + custom, bow-type dropdown + custom, distance dropdown + custom, name suggestion, override input). Below form, list of saved classes (read-only display with delete action icon).
2. **Shooting Lines Card** — Single number input for line count. Displays current value or "Nicht konfiguriert" (not configured).
3. **Rounds/Passes Card** — Radio toggle (WA Presets / Custom). If WA: three preset radio options (18m, 25m, 70m). If Custom: four number inputs (rounds, passes per round, arrows per pass, distance). Displays summary of selected config below.
4. **Presets Card** — Expandable list of saved presets with load/delete actions, plus a form to save the current setup (name input, capacity indicator, save button). Below the form, export/import buttons.

**Mobile (<768px / `md`):** Cards stack vertically, full-width, glass-treated, bottom tab bar nav applies.

**Tablet/Desktop (≥768px):** Cards may be arranged in a 2-column grid (left: Classes + Rounds/Passes; right: Shooting Lines + Presets) or remain as a single-column stack depending on content density — recommend single-column for clarity in Phase 2 planning. Sidebar nav applies.

### Registration View (Section: Schützen)

**Structure:** Top section shows mode indicator (AB / AB/CD) with explanation. Form section below: shooter name input + class dropdown + line dropdown (optional). Primary button "Schütze hinzufügen" (Add shooter). Below form, a responsive table/list of registered shooters.

**Shooter list:**
- Phone (<768px): Card-based list, one shooter per glass card, showing name + class + line (if assigned), with edit/delete icon buttons.
- Tablet/Desktop (≥768px): Formal HTML table, rows for each shooter, columns: Name, Class, Line. Edit/delete action icons in a final column.
- Glass-card styling: NOT applied to the table itself (per Phase 1 contract: tables stay opaque). The shooter list cards (on mobile) ARE glass-treated.

**Interactions:**
- Entering a shooter name + selecting a class + leaving line blank → auto-fill is set for later (in the auto-assignment modal after submission).
- Clicking "Schütze hinzufügen" → form validation (name required, class required); if valid, add shooter to table and show auto-assignment preview modal (next step).
- Auto-assignment modal (see Copywriting Contract section above) → shows how unassigned shooters will be distributed, allows trainer to review or go back and manually assign lines.
- Clicking "Speichern" in auto-assignment modal → applies assignments, closes modal, updates shooter table to show assigned lines, and optionally shows a toast "Schützen gespeichert" (Shooters saved).

### Preset Save/Load/Delete Flow

**Save:** Trainer fills in preset name input, sees capacity indicator (N of 8), clicks "Speichern". If name collision detected, a confirmation modal appears ("Vorlage X existiert bereits..."). Trainer can confirm overwrite or cancel. On success, preset is added/updated in the preset list, capacity indicator increments, and a toast appears ("Vorlage '{name}' gespeichert").

**Load:** Trainer clicks "Laden" action on a preset item. A confirmation modal appears ("Vorlage X laden?"). On confirm, the current setup (classes, lines, rounds/passes) is replaced with the preset's saved config. Shooter roster is NOT affected (per Phase 2 design). Toast confirms ("Vorlage X geladen...").

**Delete:** Trainer clicks "Löschen" action on a preset item. Destructive confirmation modal ("Vorlage X löschen?") appears. On confirm, preset is removed, capacity indicator decrements, toast confirms ("Vorlage X gelöst").

**Export all:** Button "Alle Vorlagen exportieren" → calls `dexie-export-import` to export the full database as a JSON blob, triggers a file download (filename: `presets-{ISO-DATE-TIME}.json`), toast confirms export.

**Import:** Button "Vorlagen importieren" → file input picker (accept=".json"). Trainer selects a file. On file selection, a confirmation modal appears showing how many presets will be replaced ("Import wird alle {N} aktuellen Vorlagen durch {M} importierte Vorlagen ersetzen..."). Trainer confirms → import is performed (via `dexie-export-import` with `clearTablesBeforeImport: true` for REPLACE ALL strategy). On success, preset list refreshes, capacity indicator updates, toast confirms ("Vorlagen erfolgreich importiert..."). On error, toast shows error message in red.

## Component Patterns (Svelte 5)

### Form Library & Validation

**No external form library.** Use native HTML `<select>`, `<input type="text">`, `<input type="number">`, and Svelte 5 runes (`$state`, `$derived`, `$effect`) for:
- Form state binding (e.g., `let ageGroup = $state('')`).
- Derived values (e.g., suggested class name via `$derived.by()`).
- Side effects (e.g., collision detection, persistence via `$effect`).

**Dropdown with custom escape hatch pattern:**
```svelte
<select bind:value={ageGroup}>
  <option value="">— Keine Angabe —</option>
  <option value="U12">U12</option>
  <option value="U14">U14</option>
  <option value="U16">U16</option>
  <option value="U18">U18</option>
  <option value="Erwachsene">Erwachsene</option>
  <option value="custom">— Andere —</option>
</select>
{#if ageGroup === 'custom'}
  <input type="text" placeholder="z.B. Junioren" bind:value={ageGroup} />
{/if}
```

**Class name collision detection & semantic suffix:**
When trainer saves a class, query `db.classes.where('name').equals(finalName).first()`. If collision found, append a semantic suffix based on the differing field in priority order (distance > bow-type > age-group). E.g., if "RCV-U14" exists (18m) and new class is "RCV-U14" (25m), suggest "RCV-U14-25m". No numeric or random suffixes.

**Validation approach:**
- Name field: required, non-empty, max 50 chars.
- Class dropdown: required (don't allow blank submission).
- Line assignment: optional (may be null, interpreted as auto-assign).
- Number inputs (line count, arrows/pass, etc.): required, positive integer, reasonable bounds (e.g., lines 1-10, passes 1-30, arrows 1-20).
- Display inline validation errors in red (destructive color) below the field, not in a separate error summary.

### GlassCard Reuse

Use `GlassCard.svelte` (from Phase 1) as the container for:
- Class definition form + class list.
- Shooting lines configuration.
- Rounds/passes configuration.
- Preset save/load interface and preset list.
- Shooter form and shooter list (on mobile as individual cards; on desktop, the table is NOT glass-treated but surrounded by glass-card framing).

Do NOT apply glass styling to data tables (shooter roster, class list) — keep them opaque per Phase 1 contract.

### Responsive Breakpoint

Single breakpoint: **`768px` (Tailwind `md`)** — below: mobile layout (stacked, bottom-tab nav, card-based shooter list). At/above: tablet/desktop layout (potential grid layouts if beneficial, sidebar nav, table-based shooter list).

## Modals & Overlays

### Auto-Assignment Preview Modal

Appears after trainer submits the first shooter (triggering the auto-assignment calculation). Modal is **NOT dismissible** (no X close button) — trainer must explicitly choose "Speichern" or "Zurück".

Structure:
- Overlay: semi-transparent dark overlay (e.g., `rgba(0,0,0,0.5)`) behind the modal.
- Modal container: glass-treated card (glassmorphism tokens from Phase 1), centered on screen, max-width ~520px.
- Modal title: Heading role "Schützen zugewiesen".
- Modal body: Text summary of assignment, list/grid of shooter→line mappings, rationale sentence.
- Modal footer: Two buttons (Speichern [primary/accent] and Zurück [secondary/ghost]).

### Confirmation Modals (Preset Overwrite, Delete, Load, Import)

Same glass-card treatment, centered, not dismissible, two-button footer (primary action [accent] and cancel [ghost/secondary]).

## Icons & Buttons

**Icon usage (@lucide/svelte 1.75px stroke width):**
- Shooter edit action: `Edit` icon (pencil), 20px, neutral slate color.
- Shooter delete action: `Trash2` icon (trash can), 20px, destructive red.

- Export/import: `Download` and `Upload` icons (20px, neutral).
- Form collapse/expand (if cards are collapsible): `ChevronDown` / `ChevronUp` (16px, neutral).

**Button styling:**
- Primary buttons (save, load, confirm): accent-color fill, white/light text, 44×44px minimum touch target, rounded corners (use Tailwind's `rounded-lg` or similar, consistent with Phase 1).
- Secondary/ghost buttons (cancel, back): transparent or neutral-gray outline, neutral text, same size/shape as primary.
- Destructive buttons (delete): destructive-red fill or outline, same sizing.
- Disabled state: reduced opacity (50%), `cursor: not-allowed`, no accent glow.

    importSuccess: 'Vorlagen erfolgreich importiert. {count} Vorlagen sind jetzt verfügbar.',
    importError: 'Import fehlgeschlagen: {error}',
  },

## Interaction Specifics

### Class Definition Flow

1. Trainer navigates to Setup view (Einrichtung).
2. Opens Classes card.
3. Selects age-group from dropdown (or chooses "Andere" for custom text input).
4. Selects bow-type from dropdown (or "Andere").
5. Selects distance from dropdown (or "Andere").
6. Class name is **suggested live** (via `$derived`) below the distance input, e.g., "Vorschlag: RCV-U14-18m".
7. Trainer may accept the suggestion (leave name field empty) or type a custom override.
8. Trainer clicks "Klasse hinzufügen".
9. **Collision detection:** App checks if (suggestion or override) name already exists in `db.classes`. If yes, auto-appends semantic suffix and displays the updated name as "Vorschlag: RCV-U14-18m" (conflict automatically resolved). Trainer sees the final name and can accept or re-override.
10. Class is added to `db.classes` table. List of saved classes appears below the form (read-only display, show name + delete action).
11. On delete, confirmation modal appears. On confirm, class is removed from DB and list updates.

**Key detail:** The suggested name derivation and collision detection happen **live during form edit** (via `$derived`), not only on submit. Trainer sees the final auto-suffixed name before clicking save.

### Shooting Lines Configuration

1. Simple number input (1–10 range, validation).
2. Displays current value or "Nicht konfiguriert".
3. On change, persists immediately to `db.shootingLines` table (or auto-persist via `$effect`).
4. Used in registration to compute AB/AB-CD mode.

### Rounds/Passes Configuration

1. Trainer chooses WA preset or custom config via radio buttons.
2. **If WA preset:** Three radio options (18m, 25m, 70m). On select, displays summary ("10 Passen, 3 Pfeile, 18m").
3. **If custom:** Four number inputs (rounds, passes per round, arrows per pass, distance). Displays live summary below form.
4. Persists to `db.rounds` table on save.

### Shooter Registration Auto-Assignment

1. Trainer enters shooter name + selects class + optionally sets line.
2. Clicks "Schütze hinzufügen".
3. **Validation:** name required, class required; line optional.
4. Shooter is **temporarily staged** (not yet saved to DB).
5. **Auto-assignment modal appears:**
   - Shows all shooters in the current registration session (including the new one).
   - Computes line assignments for unassigned shooters using round-robin: `(shooterIndex % lineCount) + 1` for each line, with mode-aware flights (AB vs AB/CD).
   - Displays preview: "{N} shooters will be assigned to lines 1,2,1,2,1,2,1,2,1."
   - Trainer can review and either:
     - Click "Speichern" → all shooters (assigned + auto-assigned) are persisted to `db.shooters`, auto-assignment modal closes, shooter list table updates.
     - Click "Zurück" → modal closes, auto-assignment is discarded, trainer returns to the shooter form to manually edit line assignments. Trainer may type in line numbers directly or re-submit to trigger the auto-assignment again.

**Key detail:** The auto-assignment is **transparent and reviewable** before commit, preventing confusion (Pitfall 4 from RESEARCH.md).

### Preset Import Strategy

**REPLACE ALL approach** (user answer #4):
1. Trainer clicks "Vorlagen importieren".
2. File input picker appears (accept=".json").
3. On file selection, confirmation modal appears: "Import wird alle {N} aktuellen Vorlagen durch {M} importierte Vorlagen ersetzen. Fortfahren?"
4. Trainer confirms → `dexie-export-import` is called with `clearTablesBeforeImport: true`.
5. On success, preset list refreshes, toast confirms.
6. On error, toast shows error message in red.

No selective merge or additive logic in Phase 2. REPLACE ALL is the sole strategy.
