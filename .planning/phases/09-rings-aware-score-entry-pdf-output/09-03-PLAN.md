<objective>
Thread the `rings` value from `ScoreEntry.svelte`'s existing `roundsConfig` into
`ScorePicker.svelte` (value set, color classes, keyboard map, X aria-label) and into its own
`calculatePasseSum` call, following the exact prop-threading pattern already used for
`arrowsPerPasse`. This is the UI half of TARGET-05/06/08/09 — the pure-logic contracts
(`scoreColorCategory(value, rings)`, `arrowScoreValue`/`calculatePasseSum(values, rings)`)
were already built and tested in Plan 09-01; this plan only wires them into the component
tree, converts the picker's module-level constants to props-derived `$derived.by()` values
per Svelte 5 runes reactivity rules, and fixes the hardcoded "X-Ring (10 Punkte)" aria label.

Purpose: makes the actual score-entry experience rings-correct at the range, not just the
underlying math.
Output: `ScorePicker.svelte` renders the correct button set/colors/keyboard map per rings
mode; `ScoreEntry.svelte`'s displayed row sums are rings-correct; a new
`ScorePicker.test.ts` covers both modes (closing the Wave 0 test gap flagged in research).
</objective>
