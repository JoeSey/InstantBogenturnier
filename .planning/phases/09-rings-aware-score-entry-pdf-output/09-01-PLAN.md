<objective>
Fix the TARGET-09 scoring-correctness bug and make every pure scoring/ranking function
rings-aware: `arrowScoreValue` must resolve X to 5 points under a 5-ring config (not
always 10), `scoreColorCategory` must classify 5-ring values (white/dark-blue) instead of
10-ring-only categories, and `ranking.ts` must both pass `rings` through to point-value
math and expose the raw hit counts (`count5`, `count4to1`, `countM`) that Plan 09-02's PDF
column needs. This is the foundational pure-logic wave — Plans 09-02 (PDF) and 09-03 (UI)
both build on the contracts defined here.

Purpose: DFBV (5-ring) tournaments currently silently over-count every X by 5 points in
every sum, ranking, and PDF total — this is the app's Core Value ("results ranking must
work correctly") at stake, not a cosmetic gap.
Output: Rings-aware `scoreCompletion.ts`, `scoreColor.ts`, `ranking.ts`, all with default
`rings = 10` so every existing 10-ring call site (including ones this plan doesn't touch)
keeps compiling and behaving identically until Plan 09-03 threads the real value through.
</objective>

Current `src/lib/utils/scoreCompletion.ts` (lines 6-14, exact text this plan modifies):

```typescript
export function arrowScoreValue(value: ScoreValue): number {
  if (value === 'M') return 0;
  if (value === 'X') return 10;
  return Number(value);
}

export function calculatePasseSum(values: ScoreValue[]): number {
  return values.reduce((sum, v) => sum + arrowScoreValue(v), 0);
}
```

Current `src/lib/utils/scoreColor.ts` (full file, 18 lines):

```typescript
export type ScoreColorCategory = 'yellow' | 'red' | 'blue' | 'black' | 'white' | 'miss';

export function scoreColorCategory(value: ScoreValue): ScoreColorCategory {
  if (value === 'M') return 'miss';
  if (value === 'X' || value === '10' || value === '9') return 'yellow';
  if (value === '8' || value === '7') return 'red';
  if (value === '6' || value === '5') return 'blue';
  if (value === '4' || value === '3') return 'black';
  return 'white';
}
```
