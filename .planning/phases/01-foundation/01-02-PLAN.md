<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| static host → browser | Precached shell assets served to the installed PWA; no runtime network calls. |
| service worker → running tab | A waiting SW update must not disrupt live use; activation is user-gated. |
| browser → localStorage | Session-scoped banner dismissal must NOT persist (D-03); theme (Plan 01) is the only persisted UI flag. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-04 | Tampering / DoS | UpdateBanner + registerType 'prompt' | mitigate | Banner is the load-bearing wiring for `onNeedRefresh` (Pitfall 2): without it, updates silently never activate. "Aktualisieren" gates activation on explicit user action so a mid-tournament update never force-reloads (D-01). |
| T-01-05 | Info Disclosure | Banner session dismissal | mitigate | Dismissal state kept in component/session memory only, never written to localStorage (D-03) — no persisted state that could mask a pending security-relevant update across app opens. |
| T-01-06 | Spoofing / DoS | e2e test hook to flip updateAvailable | accept | The `window.__setUpdateAvailable`-style hook is gated behind DEV/e2e only and stripped from production builds; no runtime attack surface. Acceptance criteria confirm no manual-update UI ships (D-04). |
| T-01-V14 | Configuration | Real-device install + offline launch | mitigate | ASVS V14: verified only over HTTPS/localhost (SW/install requirement); the human-verify checkpoint exercises the actual installed-PWA offline path. |
</threat_model>
